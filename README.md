# CS-230-Operating-Platforms

The Gaming Room is a client that originally developed Draw It or Lose It as an Android‑based game and wanted to expand it into a distributed, multi‑platform application. Their requirements included supporting multiple simultaneous game instances, managing players and teams, ensuring unique naming, and delivering high‑definition images rapidly across different operating systems. My role was to design a complete software architecture that could scale, remain secure, and operate consistently across Linux, Windows, macOS, and mobile platforms.

One area I did particularly well was developing clear, structured documentation that explained the architectural decisions behind the system. I translated the client’s requirements into design constraints, evaluated multiple operating platforms, and provided detailed recommendations for server architecture, storage management, memory management, distributed systems, and security. I also ensured that each section of the document connected logically to the next, making it easy for both the client and development team to understand how the system would function.

Working through the design document was especially helpful when developing the code because it forced me to think through the system’s behavior before writing any implementation. Defining the domain model, identifying constraints, and analyzing platform differences helped me anticipate challenges such as naming conflicts, memory usage, and distributed communication. This made the eventual coding process more efficient because the architectural decisions were already clear.

If I were to revise one part of my work, I would refine the System Architecture View to include more explicit diagrams or visual representations of how the server, clients, and storage interact. Although the written explanation was strong, adding a visual topology would make the architecture even easier to understand. I would improve it by incorporating a layered diagram showing the API gateway, game service, storage layer, and client interfaces.

I interpreted the user’s needs by focusing on scalability, performance, and cross‑platform compatibility. The client needed a system that could support many players at once, deliver images quickly, and maintain consistent behavior across devices. I implemented these needs by recommending Linux as the server platform, cloud object storage for images, stateless REST APIs for communication, and strong security measures such as TLS encryption and token‑based authentication. Considering user needs is essential because the success of the application depends on how well it performs for real players, not just how well it is engineered internally.

My approach to designing software involved breaking the problem into architectural layers, analyzing constraints, and evaluating platform characteristics. I relied on object‑oriented principles, distributed system patterns, and cloud‑ready design strategies to build a solution that was both scalable and maintainable. In the future, I would continue using techniques such as domain modeling, platform evaluation, and architectural prototyping to analyze and design similar applications. These strategies ensure that the final implementation aligns with user expectations, technical requirements, and long‑term scalability goals.
